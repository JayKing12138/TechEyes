下面给你一份“科技新闻雷达”**完整实现说明**，按“功能 -> 技术 -> 实现细节”展开，尽量对齐你现在这套工程（`FastAPI + 多Agent + Neo4j + Redis + PostgreSQL + hello_agents`）。

**一、系统总目标**
1. 自动获取科技新闻并结构化处理。  
2. 把“文本”变成“实体关系图谱”。  
3. 支持围绕某个实体做联合检索和持续分析。  
4. 输出可追溯、可复盘的分析结论。  

**二、功能全清单（你这个雷达系统应该包含的能力）**
1. 多源新闻采集：RSS、网站文章、搜索补充。  
2. 数据清洗与标准化：去广告、去样式、去重、语种识别。  
3. 主题归类：按公司/赛道/技术方向自动分类。  
4. 摘要生成：生成简要摘要和要点。  
5. 实体抽取：公司、人物、产品、技术、机构、地点、时间、金额。  
6. 关系抽取：投资、合作、发布、收购、任命、竞争、供应链等。  
7. 图谱写入与更新：Neo4j 增量更新、实体对齐与消歧。  
8. 热点分析：时间窗口内热词、高频实体、关系激增、事件突变。  
9. 按图索骥分析：从实体出发做本地知识库 + Web 联合检索。  
10. 历史分析与追问：保留分析上下文，支持连续问答。  
11. 质量校验与反思：Critic 校验答案可靠性，不通过则回退重试。  
12. 缓存与性能优化：多级缓存、命中短路、增量计算。  

**三、核心技术栈（按层）**
1. 服务层：`FastAPI`（接口与任务入口）。  
2. Agent层：`hello_agents`（`ReActAgent`、`SimpleAgent`），Router/Planner/Retriever/Reranker/Synthesizer/Critic 多代理协作。  
3. LLM层：兼容 OpenAI 风格 API（你项目里是可配置 provider/model/base_url）。  
4. 检索层：本地向量检索 + 关键词检索 + Web 检索（联合召回）。  
5. 图数据库：`Neo4j`（实体关系图谱、路径查询、邻域扩展）。  
6. 关系型数据库：`PostgreSQL`（原始新闻、任务记录、审计日志、分析会话）。  
7. 缓存层：`Redis + 内存 LRU + 语义缓存`（L1/L2/L3）。  
8. 异步与调度：`asyncio` + 定时任务（cron/APScheduler/Celery 皆可）。  
9. 前端图谱（若启用）：`D3.js`（交互关系图）。  
10. 观测与日志：`loguru` + 指标埋点（处理时延、命中率、失败率）。  

**四、端到端处理链路（最关键）**
1. 采集阶段：定时拉取 RSS + Web 新闻，写入 `raw_news`。  
2. 清洗阶段：正文抽取、HTML 清理、统一时间格式、内容哈希去重。  
3. 主题阶段：LLM 分类到预定义 taxonomy（如 AI芯片/大模型/机器人/政策）。  
4. 摘要阶段：生成 100-200 字摘要 + 3-5 个关键要点。  
5. 实体关系阶段：NER + 关系抽取，生成三元组 `(head, relation, tail)`。  
6. 图谱阶段：实体对齐后写入 Neo4j，边上记录来源、时间、置信度。  
7. 热点阶段：统计最近 N 小时/天实体提及量、关系变化率、跨源共现。  
8. 分析阶段：围绕目标实体检索相关文档和图谱邻域，生成结构化分析。  
9. 反思阶段：Critic 校验结论，失败则回退 Router 重新检索/生成（你已改为最多5轮）。  

**五、每个模块的实现重点与技术细节**

1. 采集模块  
- 技术：RSS 解析器 + 异步 HTTP 抓取。  
- 关键点：源站限流、失败重试、请求超时、增量拉取（按发布时间/ETag）。  
- 数据：`source_url`, `title`, `published_at`, `content_raw`, `fingerprint`。  

2. 去重与标准化模块  
- 技术：哈希去重 + 相似度去重（SimHash/embedding cosine）。  
- 关键点：标题近似重复、转载去重、同一事件多源合并。  
- 输出：`canonical_news_id`，并建立多源映射。  

3. 主题分类模块  
- 技术：LLM 分类 + 规则兜底。  
- 关键点：给定固定标签集合，强制 JSON 输出；低置信时标记 `unknown`。  
- 输出：`topic`, `subtopic`, `confidence`, `reason`。  

4. 摘要模块  
- 技术：LLM 摘要（可加引用片段）。  
- 关键点：长度控制、事实不新增、关键实体不丢失。  
- 输出：`summary_short`, `key_points[]`。  

5. 实体抽取模块  
- 技术：LLM NER + 词典/规则增强（公司后缀、型号、机构名）。  
- 关键点：中英文别名归一（例如 “OpenAI/OPEN AI”）；实体类型校正。  
- 输出：`entities[]`（name/type/aliases/confidence）。  

6. 关系抽取模块  
- 技术：LLM relation extraction + 模板约束。  
- 关键点：关系方向性、时效性、证据句定位。  
- 输出：`triples[]`（head/relation/tail/evidence/confidence/time）。  

7. 图谱构建模块（Neo4j）  
- 技术：Cypher `MERGE` 增量写入。  
- 关键点：  
- 实体主键策略：`(normalized_name, type)`。  
- 消歧策略：名称相似 + 上下文（行业、国家、关联实体）联合判定。  
- 关系属性：`source`, `first_seen`, `last_seen`, `confidence`, `count`。  
- 查询能力：邻域扩展、最短路径、关系过滤、时间切片。  

8. 热点分析模块  
- 技术：滑动时间窗统计 + 图指标（degree、betweenness、突增率）。  
- 关键点：识别“新出现但增速快”的实体；对抗单源噪声。  
- 输出：`hot_entities`, `emerging_relations`, `event_clusters`。  

9. 按图索骥分析引擎  
- 技术：图谱查询 + RAG 联合检索 + 多Agent编排。  
- 关键点：  
- 输入一个实体后，先取 Neo4j 邻域和历史关系。  
- 同时做本地文档检索与 Web 检索。  
- Reranker 统一排序，Synthesizer 产出答案，Critic 做质量校验。  
- 输出：可追溯分析（带证据引用、关系路径、历史对比）。  

10. 历史分析与追问  
- 技术：会话历史 + 项目记忆（memory）注入。  
- 关键点：保留用户关注实体、最近结论、未解问题，支持连续追问。  
- 输出：上下文连续的多轮分析结果。  

11. 质量校验与反思机制  
- 技术：Critic Agent（引用有效性 + 幻觉检测 + 证据充分性）。  
- 关键点：  
- 失败时回退 Router 重路由重检索。  
- 最大循环次数控制（你当前已设 5）。  
- 超限返回失败提示，避免输出不可靠结论。  

12. 缓存与性能  
- 技术：L1/L2/L3 缓存。  
- 关键点：  
- 相同问题 + 相似上下文命中短路。  
- 文档变更触发缓存失效。  
- 对失败结果不缓存。  

**六、你这套实现最能体现技术能力的“亮点”**
1. 不是单点功能，而是多模块闭环系统设计能力。  
2. 能把 LLM、检索、图数据库、缓存、调度整合成生产链路。  
3. 处理了真实工程难点：去重、消歧、时效、质量控制、可追溯。  
4. 引入 Critic 反思回路，体现“可靠性优先”的 AI 工程思维。
